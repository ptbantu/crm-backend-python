## 📋 **更新后的业务需求文档整理**

### 1. 客户信息与管理

| 序号 | 需求点 | 描述/说明 |
| :--- | :--- | :--- |
| 1.1 | **客户来源渠道划分** | 需要在系统中对客户的来源渠道进行明确划分和记录。 |
| 1.2 | **客户对接人字段信息** | 记录客户对接人的详细信息，包括**年龄、性别、喜好**及其他相关信息。 |
| 1.3 | **大客户/全包标识** | 区分客户类型，例如：**Eason（大客户），Lily（全包）**。 |
| 1.4 | **客户对内/对外负责人** | 确定客户内部的主要负责人（客户对内），以及公司内部的Bantu对客户的首要负责人（内部的Bantu对客户的首要负责人）。 |
| 1.5 | **客户归属** | 划分客户归属类型：**公司直接获得（如老板）**的客户；**员工自己创建**的客户（需KK创建和出单）。 |
| 1.6 | **线索/客户转移节点** | **线索不能录入**，客户归属原则为“谁拿到客户归谁”（例如：老板申请后，可将Eason从小型销售转移给其他人）。 |
| 1.7 | **客户/联系人逻辑关系** | 客户之间存在**1对多**的逻辑关系；联系人可作为父集，区分联系人A和B作为主要负责人。 |

### 2. 订单主体与业务逻辑

| 序号 | 需求点 | 描述/说明 |
| :--- | :--- | :--- |
| 2.1 | **订单主体** | 订单主体是**综合单-销售单**；订单归属到**客户**，订单核算可**以客户为主体或以公司为主体**。 |
| 2.2 | **综合订单拆分** | **综合订单A**可包含**项目1-项目5**等多个子项目。 |
| 2.3 | **协作人员跟进** | 允许邀请**协作人员**（如销售助理）一起跟进大客户的小部分订单。 |
| 2.4 | **销售助理协作分润** | 如果邀请销售助理协作，需要设计**销售的利润分润区间**并确定**修改规则**。 |
| 2.5 | **业务推进与核算** | **核心需求：** 算出**利润**和**业务推进**为主；第一版需求以**业务推进**（业务文档和交互信息）为主。 |

### 3. 销售与执行（中台）团队整合

| 序号 | 需求点 | 描述/说明 |
| :--- | :--- | :--- |
| 3.1 | **执行组整合** | 整合执行组为：**1. 企业服务组, 2. 签证, 3. 财税, 4. 猎头组**，作为**完全执行**团队。 |
| 3.2 | **中台执行人确定** | **KK（中台执行角色）**负责确定执行人，需考虑**速度、利润分配、执行周期长度**等因素进行选择。 |
| 3.3 | **员工可见信息** | 员工需要能看到：**考勤、报销、提成**，以及**个人提成明细**。 |
| 3.4 | **内部财务动作** | 内部财务角色需负责**支付**和**开票**动作。 |

### 4. 销售/执行人员利润与提成体系

| 序号 | 需求点 | 描述/说明 |
| :--- | :--- | :--- |
| 4.1 | **毛利计算公式** | $$毛利 = 收入 - 供应商成本 - 浮动成本 - 销售人员的成本$$ |
| 4.2 | **浮动成本核算** | **浮动成本**以**报销形式支出**（如路费、客户招待），需跟**具体订单任务项**挂钩。中台浮动成本需跟**具体的业务挂钩**（如员工外出费用）。 |
| 4.3 | **销售利润划分周期** | 确定销售利润是**按月划分**还是**按每单划分**。 |
| 4.4 | **中台执行人员提成** | 执行线提成方式：**1. 固定金额/每件**；**2. 按照订单比率划分**；服务设置成**固定划分**，销售人员使用**固定金额**的提成。 |
| 4.5 | **中台利润核算** | 中台利润需要**Follow 1（毛利）**，然后**减掉中台的浮动成本**。 |
| 4.6 | **提成计算时间** | 订单**完成**（订单达到**交付状态**）之后才开始计算成本和提成。 |
| 4.7 | **大订单分润模式** | 考虑大订单模式的客户分润比例（例如：**7/3**）。 |
| 4.8 | **订单价格调整** | 如果订单起初价格（如10w）之后因客户砍价减少，提成等需**等比例减少**。 |
| 4.9 | **折扣底线** | 需要明确设置**折扣价的底线**（有哪些）。 |

### 5. 销售和价格体系表

| 序号 | 需求点 | 描述/说明 |
| :--- | :--- | :--- |
| 5.1 | **价格体系档次** | 售价录入系统，以**注册**为例，分为至少**4-5档**：**直客价格**、**渠道价格A（大客户）**、**渠道价格B（销售渠道）**、**特殊价格**。 |
| 5.2 | **价格生效时间** | 设置一个**价格生效的时间节点**（例如：基础为**24小时**）。 |
| 5.3 | **客户等级对应价格** | 客户按照**等级**对应不同的**价格体系**。 |
| 5.4 | **供应商成本确认** | **供应商的成本**和**价格生效时期**需要确认。 |

### 6. 项目执行进度与时效管理（新增）

| 序号 | 需求点 | 描述/说明 |
| :--- | :--- | :--- |
| 6.1 | **标准执行时长设置** | 每个**单独服务项**需设置**标准执行时长**（例如：7天）。 |
| 6.2 | **执行阶段划分** | 标准执行时长需划分为多个阶段，包括：**资料收集中**、**任务执行中**、**提交完成等待审批**、**任务已完成**。 |
| 6.3 | **阶段细分期限** | 每个阶段需要设置**细分的时间期限**。 |
| 6.4 | **进度预警机制** | 需要设置**预警**机制，用于提示即将或已超期的任务。 |
| 6.5 | **预期与实际追踪** | 系统需追踪和显示**预期**完成时间和**实际**完成时间。 |
